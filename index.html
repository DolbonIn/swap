Part 1: The Old Limit

The Core Limitation of JSON-based Workflows

The existing JSON method may be clear to humans, but to a model, it's just a list of keys and values.
This becomes a major cause of hindering logical reasoning and increasing error rates.

Why is JSON Difficult for Models?

Models had to memorize unfamiliar 'rules' rather than the structural meaning of JSON.

JSON-based Thinking (Memorization-centric)

It's difficult for the model to infer connections between nodes and data flow. It must memorize new key names, formats, and linking rules like 'nodes', 'links', 'params'. This process often leads to structural errors or unnecessary references.

Part 2: The New Approach

Python DSL: Making Models 'Think Like Code'

Python DSL expresses the workflow's execution process in a form that allows 'thinking like a program'. Within this structure, the model can intuitively understand logical relationships and data flow.

JSON vs. Python DSL: The Difference in Intuitiveness

Comparing the same 'Hotel Booking' logic in both formats.

Existing JSON Method (Data Listing)

{
  "nodes": [
    {"id": 1, "type": "get_hotel_info", "params": {"city": "Paris"}},
    {"id": 2, "type": "book_hotel", "params": {"hotel": "@1.hotel"}}
  ],
  "links": [{"src": 1, "dst": 2, "field": "hotel"}]
}

Python DSL Method (Logical Flow)

with Flow("hotel_booking") as f:
    hotels = op.get_hotel_info(city="Paris")
    op.book_hotel(hotel=hotels.selected)

Part 3: Core Principle

Core Principle: Learning Efficiency from 'Familiarity'

DSL-based GRPO simultaneously removes the 'burden of memorizing new things' and the 'uncertainty of execution' for the model.

Understanding-based Learning

Models are already familiar with Python syntax from pre-training. The DSL operates on top of this familiar environment. Instead of memorizing unfamiliar rules, the model can immediately interpret reward signals and correct its behavior within a linguistic context it understands.
